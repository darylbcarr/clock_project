  ---                                                                                                                                                                      
  How Versioning, Commits, and Releases Relate                                                                                                                             
                                                                                                                                                                           
  The Version Number Lives in One Place                                                                                                                                    
                                           
  CMakeLists.txt → set(PROJECT_VER "1.1.2")                                                                                                                                
                                                                                                                                                                           
  ESP-IDF bakes this into the firmware binary as app_desc.version. OtaManager::running_version() reads it at runtime via esp_app_get_description(). The OTA checker        
  compares it against version.json on GitHub to decide if an update is available.                                                                                          
                                                                                 
  ---                                                                                                                                                                      
  The Three-Step Workflow                  
                         
  code changes  →  commit(s)  →  bump version + tag  →  push tag  →  CI builds + releases                                                                                  
                                                                                         
  1. Regular development commits — commit as often as you like, no version change needed. These are just work in progress on main.                                         
                                                                                                                                                                           
  2. When ready to release — do this in a single commit:                                                                                                                   
  # Edit CMakeLists.txt: bump PROJECT_VER "1.1.2" → "1.1.3"                                                                                                                
  git add CMakeLists.txt                                   
  git commit -m "Bump to v1.1.3"                                                                                                                                           
  git tag v1.1.3                                                                                                                                                           
  git push origin main v1.1.3                                                                                                                                              
                                                                                                                                                                           
  Pushing the tag triggers the GitHub Actions workflow, which builds, creates a GitHub Release with clock_project.bin, and updates version.json on main. Devices polling   
  for OTA will see the new version within their next check interval.                                                                                                       
                                                                                                                                                                           
  3. The tag is the source of truth — the tag name must match PROJECT_VER. If they differ, the OTA check will never trigger (running version won't match version.json).

● Clean. To summarise the full behaviour now:

  Every minute — advance_one_minute() moves the hand exactly one minute forward. No sensor involved.

  Near the top of each hour (within ±30s of :00) — instead of a fixed advance, check_sensor_and_correct() takes over:
  - Advances 10 steps at a time, reading the sensor between each chunk
  - Slot found → advances sensor_offset_steps_ more → hand lands at exactly 12:00. Displayed position snapped to HH:00.

  cmd_set_time() handles that drift.

  Large corrections — handled by SNTP. Whenever the network time syncs, cmd_set_time() fast-forwards the hand to match real time.

  The step-by-step scan at 10 steps per check takes about 1.2 seconds when the slot is at the very end of the minute range — imperceptible to a viewer.

● WiFi and Matter are independent layers with an overlap only during setup:                                                                                                    
                                                                                                                                                                                 
  WiFi is the network transport — used for SNTP time sync, web interface, geolocation, and OTA updates. The clock needs WiFi to function as designed.                            
                                                                                                                                                                                 
  Matter is a smart home commissioning and control protocol. Its only relationship to WiFi is:                                                                                   
  - During the Matter setup path: Matter uses BLE to securely deliver WiFi credentials to the device (that's the whole point of BLE commissioning — the smart home app pushes
  SSID/password over BLE)                                                                                                                                                        
  - After commissioning: Matter runs on top of WiFi to expose the device to Alexa/Apple Home/Google Home
                                                                                                                                                                                 
  If the user chooses WiFi (manual entry), Matter does NOT need to run for the core clock function:                                                                              
  - Time sync (SNTP) ✓ — pure WiFi                                                                                                                                               
  - Web interface ✓ — pure WiFi                                                                                                                                                  
  - OTA updates ✓ — pure WiFi                                                                                                                                                    
                                                                                                                                                                                 
  Matter is only needed if the user wants smart home integration (voice control, automations, etc.). A user who chose the WiFi path explicitly bypassed Matter commissioning and
  may not want or use smart home features at all.                                                                                                                                
  
  Currently the firmware always calls s_matter.start() for WiFi-path devices. This:                                                                                              
  - Wastes ~100KB RAM on the Matter stack  
  - Starts BLE advertising (which caused the connection failure you just hit)                                                                                                    
  - Adds complexity for users who just want a standalone clock               
                                                                                                                                                                                 
  The cleanest design would be to track which setup path was used in NVS and only start Matter if commissioned or explicitly requested. But that's a larger design change.       
                                                                                                                                                                                 
  For now, the deferred-start fix prevents the coex failure. A future step would be a "Skip Matter" option on the first-time setup screen for users who don't want smart home

  How CHIP Stores WiFi Credentials                                                                                                                                               
                                                                                                                                                                                 
  In NetworkCommissioningDriver.cpp line 98-111, ESPWiFiDriver::Init() at boot does this:                                                                                        
                                                                                                                                                                                 
  esp_wifi_get_config(WIFI_IF_STA, &stationConfig)                                                                                                                               
  // loads stationConfig.sta.ssid/password into mStagingNetwork                                                                                                                  
                                                                                                                                                                                 
  CHIP reads credentials from the WiFi driver NVS (esp_wifi_get_config), not from CHIP_KVS. The wifi-ssid/wifi-pass keys in CHIP_KVS are a temporary fail-safe backup that gets  
  deleted by CommitConfiguration() after a successful commissioning. They're normally absent.                                                                                    
                                                                                                                                                                                 
  This means updating WiFi credentials for a Matter device is clean: just call esp_wifi_set_config() with the new credentials and CHIP picks them up at next boot — no CHIP_KVS  
  surgery needed, fabric stays intact.
                                                                                                                                                                                 
  ---                                      
  Both Modes Can Be Updated
                           
  WiFi-only: Save new SSID/password to NetCfg NVS → restart. Our code uses it directly.
                                                                                                                                                                                 
  Matter: Call esp_wifi_set_config(WIFI_IF_STA, &new_cfg) with new credentials (this updates the WiFi driver NVS) → restart. CHIP's ESPWiFiDriver::Init() reads the new          
  credentials from there, connects normally. Fabric/commissioning data untouched.                                                                                                
                                                                                                                                                                                 
  The two paths differ slightly in where credentials are stored, but the user entry flow via show_wifi_credentials() is identical for both.
