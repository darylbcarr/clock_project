  ---                                                                                                                                                                      
  How Versioning, Commits, and Releases Relate                                                                                                                             
                                                                                                                                                                           
  The Version Number Lives in One Place                                                                                                                                    
                                           
  CMakeLists.txt → set(PROJECT_VER "1.1.2")                                                                                                                                
                                                                                                                                                                           
  ESP-IDF bakes this into the firmware binary as app_desc.version. OtaManager::running_version() reads it at runtime via esp_app_get_description(). The OTA checker        
  compares it against version.json on GitHub to decide if an update is available.                                                                                          
                                                                                 
  ---                                                                                                                                                                      
  The Three-Step Workflow                  
                         
  code changes  →  commit(s)  →  bump version + tag  →  push tag  →  CI builds + releases                                                                                  
                                                                                         
  1. Regular development commits — commit as often as you like, no version change needed. These are just work in progress on main.                                         
                                                                                                                                                                           
  2. When ready to release — do this in a single commit:                                                                                                                   
  # Edit CMakeLists.txt: bump PROJECT_VER "1.1.2" → "1.1.3"                                                                                                                
  git add CMakeLists.txt                                   
  git commit -m "Bump to v1.1.3"                                                                                                                                           
  git tag v1.1.3                                                                                                                                                           
  git push origin main v1.1.3                                                                                                                                              
                                                                                                                                                                           
  Pushing the tag triggers the GitHub Actions workflow, which builds, creates a GitHub Release with clock_project.bin, and updates version.json on main. Devices polling   
  for OTA will see the new version within their next check interval.                                                                                                       
                                                                                                                                                                           
  3. The tag is the source of truth — the tag name must match PROJECT_VER. If they differ, the OTA check will never trigger (running version won't match version.json).

● Here's the problem clearly laid out:

  Current flow (broken):
  1. advance_one_minute() — motor moves the hand forward one full minute
  2. Then check_sensor_and_correct() — reads sensor at the new position

  The slot is just before 12:00. By the time the sensor reads, the hand has already been advanced past the slot. It will almost never trigger.

  The only scenario where it would trigger is if the clock was slow enough that after the advance the hand happens to land exactly at the slot — a very narrow
  coincidence.

  ---
  What the design should do — near the hour, replace the fixed 1-minute advance with a step-by-step scan that watches the sensor:

  - Clock is slow (hand short of slot): scan finds the slot, advances offset steps → exact 12:00 ✓
  - Clock is on time: scan finds the slot at the expected position → exact 12:00 ✓
  - Clock is fast (hand already past the slot): scan doesn't find it, falls back to a normal 1-minute advance; SNTP handles the larger drift ✓

● Clean. To summarise the full behaviour now:

  Every minute — advance_one_minute() moves the hand exactly one minute forward. No sensor involved.

  Near the top of each hour (within ±30s of :00) — instead of a fixed advance, check_sensor_and_correct() takes over:
  - Advances 10 steps at a time, reading the sensor between each chunk
  - Slot found → advances sensor_offset_steps_ more → hand lands at exactly 12:00. Displayed position snapped to HH:00.
  - Slot not found in one minute of travel → completes a normal 1-minute advance. This happens when the clock is running fast (hand already past the slot). SNTP's
  cmd_set_time() handles that drift.

  Large corrections — handled by SNTP. Whenever the network time syncs, cmd_set_time() fast-forwards the hand to match real time.

  The step-by-step scan at 10 steps per check takes about 1.2 seconds when the slot is at the very end of the minute range — imperceptible to a viewer.
